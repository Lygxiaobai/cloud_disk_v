本文档使用纯 Markdown 和 ASCII 流程图描述本次新增功能的核心流程，适合在不支持 Mermaid 的 Markdown 预览器中直接查看。

## 一、文件上传流程

### 1. 功能目标

文件上传功能支持以下能力：

- 秒传
- 分片上传
- 断点续传
- 暂停 / 继续
- 失败重试
- 上传进度展示
- 剩余时间估算
- STS 临时授权上传

### 2. 设计思路

上传流程拆成三段：

- 第一段：上传初始化 `init`
- 第二段：前端直传 OSS
- 第三段：上传完成确认 `complete`

这样做的好处是：

- 文件内容不经过业务服务器，减轻后端压力
- 可以充分利用 OSS 的分片上传能力
- 前端可以天然做进度、暂停、继续、重试
- 后端可以统一控制权限和 objectKey

### 3. 流程图

```text
+------------------------+
| 用户选择本地文件       |
+------------------------+
            |
            v
+------------------------+
| 前端计算文件 MD5       |
+------------------------+
            |
            v
+-----------------------------+
| 调用 POST /file/upload/init |
+-----------------------------+
            |
            v
+----------------------------------+
| 后端根据 hash + size 查询资源池  |
+----------------------------------+
      | 命中                               | 未命中
      v                                    v
+---------------------------+     +---------------------------+
| 直接写入 user_repository  |     | 创建 upload_session      |
+---------------------------+     +---------------------------+
      |                                    |
      v                                    v
+---------------------------+     +---------------------------+
| 返回 instant_hit = true   |     | 生成 objectKey           |
+---------------------------+     +---------------------------+
      |                                    |
      v                                    v
+---------------------------+     +---------------------------+
| 上传任务直接结束          |     | 调用 STS AssumeRole      |
+---------------------------+     +---------------------------+
                                           |
                                           v
                              +-------------------------------+
                              | 返回 STS + sessionIdentity    |
                              | + objectKey                   |
                              +-------------------------------+
                                           |
                                           v
                              +-------------------------------+
                              | 前端使用 ali-oss 直传 OSS     |
                              +-------------------------------+
                                           |
                                           v
                              +-------------------------------+
                              | 上传过程中自动刷新 STS        |
                              | 或手动暂停 / 继续             |
                              +-------------------------------+
                                           |
                                           v
                              +-------------------------------+
                              | 上传完成后调用                |
                              | POST /file/upload/complete    |
                              +-------------------------------+
                                           |
                                           v
                              +-------------------------------+
                              | 后端校验对象大小              |
                              | 写 repository_pool            |
                              | 写 user_repository            |
                              | 更新 upload_session 状态      |
                              +-------------------------------+
                                           |
                                           v
                              +-------------------------------+
                              | 前端显示上传完成              |
                              +-------------------------------+
```

## 二、文件预览流程

### 1. 功能目标

文件预览支持以下几类：

- 图片
- 视频
- 音频
- PDF
- 文本类文件
- 其他文件下载打开

### 2. 设计思路

不同文件类型采用不同方案：

- 图片 / 视频 / 音频 / PDF：返回 OSS 临时签名 URL，前端直接展示
- 文本类：后端只读取前 4KB 内容，避免大文件直接全部加载
- 其他类型：返回下载地址

这样既能控制权限，又能兼顾性能。

### 3. 流程图

```text
+------------------------+
| 用户点击文件预览       |
+------------------------+
            |
            v
+-----------------------------------+
| 调用 GET /user/file/preview/:id   |
+-----------------------------------+
            |
            v
+-------------------------------+
| 后端根据扩展名判断文件类型    |
+-------------------------------+
     | 图片/视频/音频/PDF
     |-------------------------------> 生成 OSS 签名 URL
     |
     | 文本
     |-------------------------------> 读取前 4KB 文本
     |
     | 其他
     |-------------------------------> 返回下载地址
            |
            v
+------------------------+
| 前端弹窗展示预览       |
+------------------------+
```

## 三、最近文件流程

### 1. 功能目标

记录用户最近访问过的文件，方便快速查看。

### 2. 设计思路

最近文件不是单独维护一张数据库表，而是使用 Redis 的有序集合：

- key：`cloud_disk:recent:{userIdentity}`
- member：文件 identity
- score：最后访问时间戳

优点：

- 同一个文件可自动覆盖旧记录
- 天然支持按最近时间排序
- 查询效率高

### 3. 流程图

```text
+------------------------+
| 用户成功预览文件       |
+------------------------+
            |
            v
+-------------------------------+
| 后端写 Redis ZSet 最近记录    |
+-------------------------------+
            |
            v
+-----------------------------+
| 调用 GET /user/file/recent  |
+-----------------------------+
            |
            v
+------------------------+
| 前端展示最近文件       |
+------------------------+
```

## 四、搜索 / 筛选 / 排序流程

### 1. 功能目标

支持：

- 文件名搜索
- 文件类型筛选
- 收藏筛选
- 按名称 / 大小 / 时间排序

### 2. 设计思路

不额外新增多个接口，而是扩展原有文件列表接口：

- `query`
- `file_type`
- `favorite_only`
- `order_by`
- `order_dir`

后端动态拼接查询条件，前端统一维护筛选状态。

### 3. 流程图

```text
+------------------------+
| 用户输入搜索条件       |
| 或切换筛选排序        |
+------------------------+
            |
            v
+----------------------------+
| 调用 GET /user/file/list   |
+----------------------------+
            |
            v
+-------------------------------+
| 后端按条件动态构造 SQL       |
| - query                      |
| - file_type                  |
| - favorite_only              |
| - order_by / order_dir       |
+-------------------------------+
            |
            v
+------------------------+
| 返回列表结果           |
+------------------------+
            |
            v
+------------------------+
| 前端刷新表格           |
+------------------------+
```

## 五、收藏流程

### 1. 功能目标

支持单个收藏和批量收藏。

### 2. 设计思路

收藏采用最轻量实现：

- 直接在 `user_repository` 表增加 `is_favorite`
- 不单独拆新表

这样查询和筛选都非常直接。

### 3. 流程图

```text
+------------------------+
| 用户点击收藏按钮       |
| 或批量收藏            |
+------------------------+
            |
            v
+--------------------------------+
| 调用收藏接口更新 is_favorite   |
+--------------------------------+
            |
            v
+------------------------+
| 刷新当前文件列表       |
+------------------------+
```

## 六、批量操作流程

### 1. 功能目标

支持：

- 批量收藏
- 批量移动
- 批量删除

### 2. 设计思路

批量操作的关键点是：

- 文件夹操作不能只处理当前节点
- 必须带上整个子树
- 必须使用事务保证一致性

### 3. 流程图

```text
+------------------------+
| 用户勾选多个文件       |
+------------------------+
            |
            v
+------------------------+
| 选择批量操作类型       |
+------------------------+
    | 批量收藏
    |-----------------------------> 更新 is_favorite
    |
    | 批量移动
    |-----------------------------> 校验目标目录
    |                               校验不能移动到自己或子目录
    |                               批量更新 parent_id
    |
    | 批量删除
    |-----------------------------> 收集整棵子树
                                    批量写 deleted_at
```

## 七、回收站流程

### 1. 功能目标

支持：

- 删除进入回收站
- 回收站查看
- 恢复
- 彻底删除

### 2. 设计思路

回收站基于软删除实现：

- 普通删除：写 `deleted_at`
- 恢复：清空 `deleted_at`
- 彻底删除：物理删除数据库记录

这里还有一个额外业务约束：

- 如果父目录还在回收站，不能先恢复子文件

### 3. 流程图

```text
+------------------------+
| 用户删除文件 / 文件夹  |
+------------------------+
            |
            v
+------------------------+
| 写 deleted_at          |
| 进入回收站             |
+------------------------+
            |
            v
+-----------------------------+
| 调用 GET /user/recycle/list |
+-----------------------------+
            |
            v
+------------------------+
| 前端展示回收站         |
+------------------------+
      | 恢复                           | 彻底删除
      v                                v
+---------------------------+   +---------------------------+
| 检查父目录是否已恢复      |   | 物理删除数据库记录        |
+---------------------------+   +---------------------------+
      | 是        | 否
      v           v
+-----------+   +------------------------+
| 清空      |   | 拒绝恢复子文件         |
| deleted_at|   +------------------------+
+-----------+
```

## 八、整体总结

本次新增功能的整体特点如下：

- 上传采用“前端直传 OSS + 后端发 STS”的方案，性能更好
- 秒传通过 `hash + size` 命中资源池实现
- 断点续传依赖 `upload_session + checkpoint`
- 预览采用“签名 URL + 文本分段读取”的组合方案
- 最近文件采用 Redis ZSet 实现
- 收藏采用字段方案实现，轻量直接
- 批量操作和回收站统一基于子树处理和事务控制

这套设计既能保证功能完整，也保留了后续继续扩展的空间。
