## 一、安全类 — 紧急修复（P0）

> 这些问题存在严重安全隐患，必须在上线前修复。

### 1.1 密码哈希算法替换

**现状：** 密码使用纯 MD5 哈希，无盐值（`core/internal/helper/helper.go:30`）。MD5 已被彻底破解，彩虹表可在秒级内逆向大部分密码。

**方案：**
- 将 `MD5(password)` 替换为 `bcrypt.GenerateFromPassword(password, bcrypt.DefaultCost)`
- 登录验证改用 `bcrypt.CompareHashAndPassword`
- 编写数据迁移脚本：首次部署时强制所有用户重置密码，或在下次登录时用旧 MD5 验证后自动升级为 bcrypt 存储
- 引入库：`golang.org/x/crypto/bcrypt`

**涉及文件：**
- `core/internal/helper/helper.go` — 新增 `HashPassword` / `CheckPassword`
- `core/internal/logic/user-register-logic.go:97` — 注册时调用新哈希
- `core/internal/logic/user-login-logic.go:38` — 登录时用 bcrypt 比对

---

### 1.2 JWT 密钥与敏感信息外部化

**现状：** JWT 签名密钥硬编码为 `"cloud-disk-key"`（`core/internal/define/define.go:14`），邮箱密码同样硬编码。数据库密码 `root:123456`、RabbitMQ `guest:guest` 在配置文件中明文存储。

**方案：**
- JWT 密钥改为随机生成的 256 位字符串，通过环境变量 `JWT_SECRET` 注入
- 邮箱密码、数据库密码、RabbitMQ 密码、OSS 凭证全部迁移到环境变量
- `core-api.yaml` 中使用 `${ENV_VAR}` 占位符，go-zero 支持环境变量替换
- 生产环境部署文档中标注所有必需环境变量
- Access Token 和 Refresh Token 使用不同的签名密钥，防止令牌混用

**涉及文件：**
- `core/internal/define/define.go` — 删除硬编码，读取环境变量
- `core/etc/core-api.yaml` — 敏感值替换为环境变量引用
- 新增 `.env.example` — 列出所有必需环境变量模板

---

### 1.3 接口限流

**现状：** 所有接口无限流保护（`core/internal/handler/routes.go`），登录/注册/验证码发送接口可被无限调用。

**方案：**
- 使用 go-zero 内置的 `rest.WithMaxBytes()` 限制请求体大小
- 针对敏感接口增加 IP 级别限流中间件（基于 Redis 的滑动窗口）：
  - `POST /user/login` — 同一 IP 每分钟最多 10 次
  - `POST /mail/code/send/register` — 同一邮箱每分钟 1 次，每小时 5 次
  - `POST /user/register` — 同一 IP 每小时 20 次
- 登录连续失败 5 次后锁定账户 15 分钟（Redis 计数）
- 返回标准 `429 Too Many Requests` 响应与 `Retry-After` 头

**涉及文件：**
- 新增 `core/internal/middleware/rate-limit-middleware.go`
- `core/internal/handler/routes.go` — 为指定路由挂载限流中间件

### 1.4 验证码泄露修复

**现状：** 邮箱验证码直接在 API 响应中返回（`core/internal/logic/mail-code-send-register-logic.go:64`），客户端无需查看邮件即可拿到验证码，邮箱验证形同虚设。

**方案：**
- 删除 `MailCodeResponse.Code` 字段
- 响应改为 `{ "message": "验证码已发送到您的邮箱" }` 
- 前端隐藏验证码展示逻辑

**涉及文件：**
- `core/internal/types/types.go` — 删除 `Code` 字段
- `core/internal/logic/mail-code-send-register-logic.go:64` — 不返回验证码
- `web/src/views/auth/RegisterView.vue` — 移除验证码显示

### 1.5 错误信息脱敏

**现状：** 内部错误（数据库连接失败、文件路径等）直接返回给客户端（`core/internal/errors/logic_error.go:17`）。Auth 中间件、Casbin 中间件同样泄露内部错误。

**方案：**
- `LogicError.Error()` 仅返回用户友好的 `message` 字段，内部 `err` 仅写入日志
- Auth 中间件统一返回 `{ "code": 401, "message": "认证失败" }`
- Casbin 中间件统一返回 `{ "code": 403, "message": "权限不足" }`
- 生产环境下所有 500 错误统一返回 `{ "code": 500, "message": "服务器内部错误" }`

**涉及文件：**
- `core/internal/errors/logic_error.go` — `Error()` 不拼接内部错误
- `core/internal/middleware/auth-middleware.go:34` — 通用错误响应
- `core/internal/middleware/casbin-middleware.go:28-29` — 通用错误响应
- `core/internal/middleware/error-recovery-middleware.go` — panic 恢复不暴露堆栈

### 1.6 文件上传安全校验

**现状：** 上传接口无文件类型白名单、无文件大小上限、无 MIME 类型检测（`core/internal/handler/file-upload-handler.go`）。

**方案：**
- 新增配置项：`MaxUploadSize`（默认 2GB）、`AllowedExtensions`（可配置白名单）
- STS 上传初始化时校验文件扩展名
- 上传完成回调时通过 `HeadObject` 校验实际文件大小是否与声明一致（已有此逻辑，保持）
- 旧版直传接口增加 `http.MaxBytesReader` 限制请求体大小
- 禁止上传可执行文件（.exe/.bat/.sh/.cmd 等）

**涉及文件：**
- `core/etc/core-api.yaml` — 新增上传配置项
- `core/internal/logic/upload-init-logic.go` — 扩展名校验
- `core/internal/handler/file-upload-handler.go` — 增加大小限制
- `core/internal/handler/file-upload-multipart-handler.go` — 增加大小限制

### 1.7 CORS 配置

**现状：** 无 CORS 配置，前后端分离部署时接口不可用。

## 二、性能优化（P1）

### 2.1 数据库索引（最高优先级）

**现状：** 整个项目未定义任何数据库索引。所有查询都是全表扫描，数据量增长后性能将急剧下降。

**方案：** 在 xorm 模型结构体中添加索引标签，或编写 SQL 迁移脚本：

```sql
-- user_repository 表（查询最频繁）
CREATE INDEX idx_ur_user_identity ON user_repository(user_identity);
CREATE INDEX idx_ur_parent_id ON user_repository(parent_id);
CREATE INDEX idx_ur_identity ON user_repository(identity);
CREATE INDEX idx_ur_deleted_at ON user_repository(deleted_at);
CREATE INDEX idx_ur_user_parent_deleted ON user_repository(user_identity, parent_id, deleted_at);
CREATE INDEX idx_ur_user_deleted ON user_repository(user_identity, deleted_at);

-- repository_pool 表
CREATE UNIQUE INDEX idx_rp_identity ON repository_pool(identity);
CREATE INDEX idx_rp_hash_size ON repository_pool(hash, size);

-- share_basic 表
CREATE UNIQUE INDEX idx_sb_identity ON share_basic(identity);
CREATE INDEX idx_sb_user_identity ON share_basic(user_identity);

-- upload_session 表
CREATE UNIQUE INDEX idx_us_identity ON upload_session(identity);
CREATE INDEX idx_us_user_status ON upload_session(user_identity, status);

-- user_file_version 表
CREATE INDEX idx_ufv_file_identity ON user_file_version(file_identity);

-- user_basic 表
CREATE UNIQUE INDEX idx_ub_identity ON user_basic(identity);
CREATE UNIQUE INDEX idx_ub_name ON user_basic(name);
CREATE UNIQUE INDEX idx_ub_email ON user_basic(email);
```

**涉及文件：**
- `core/internal/models/*.go` — 添加 xorm 索引标签
- 新增 `deploy/sql/migrations/001_add_indexes.sql`

---

### 2.2 数据库连接池配置

**现状：** xorm 和 Redis 均使用默认配置（`core/internal/models/init.go`）。MySQL 默认仅 2 个空闲连接且不限最大连接数，高并发时会耗尽数据库连接。

**方案：**
```go
// MySQL
engine.SetMaxIdleConns(10)
engine.SetMaxOpenConns(100)
engine.SetConnMaxLifetime(time.Hour)

// Redis
redis.NewClient(&redis.Options{
    PoolSize:     20,
    MinIdleConns: 5,
    DialTimeout:  5 * time.Second,
    ReadTimeout:  3 * time.Second,
    WriteTimeout: 3 * time.Second,
    MaxRetries:   3,
})
```

**涉及文件：**
- `core/internal/models/init.go` — Init 和 InitRedis 函数

---

### 2.3 N+1 查询优化

**现状与位置：**

| 问题 | 文件 | 说明 |
|------|------|------|
| 最近文件逐条查询 | `user-recent-file-list-logic.go:45-82` | 50 条 Redis 记录触发 50 次 SQL 查询 |
| 子树收集逐层 BFS | `file_common.go:159-198` | 深层目录多次递归查询 |
| 批量移动逐条验证 | `user-file-batch-move-logic.go:64-104` | N 个文件 ≥ 2N 次查询 |
| 批量重命名逐条处理 | `user-file-batch-rename-logic.go:66-98` | 每个文件 3 次查询 |
| 文件夹路径逐层上溯 | `user-folder-path-logic.go:59-83` | 10 层深度 = 10 次查询 |

**方案：**
- **最近文件：** 改为 `WHERE identity IN (...)` 批量查询，应用层按 Redis 排序还原顺序
- **子树收集：** 使用 MySQL 8.0 递归 CTE `WITH RECURSIVE` 一次性查询整棵子树
- **批量移动/重命名：** 先批量查询所有涉及文件，在内存中校验后批量更新
- **文件夹路径：** 使用递归 CTE 一次性查询从目标到根的路径

---

### 2.4 旧版上传内存溢出修复

**现状：** `file-upload-handler.go:31` 将整个文件读入内存（`make([]byte, fileHeader.Size)`），无大小限制。`file-upload-multipart-handler.go:34` 同样使用 `io.ReadAll`。

**方案：**
- 使用 `io.LimitReader` 限制读取大小
- 文件哈希计算改为流式（`io.Copy` + `md5.New()`），不将整个文件加载到内存
- OSS 上传使用 `io.Reader` 流式传输
- 建议逐步废弃旧版上传接口，全面迁移到 STS 直传模式

**涉及文件：**
- `core/internal/handler/file-upload-handler.go`
- `core/internal/handler/file-upload-multipart-handler.go`
- `core/internal/helper/helper.go` — FileUpload / FileUploadMultipart

---

### 2.5 缓存击穿防护

**现状：** 热门分享详情缓存过期后（1 小时 TTL），大量并发请求同时穿透到数据库（`core/internal/cache/share_cache.go:101-122`）。

**方案：**
- 引入 `golang.org/x/sync/singleflight`
- 缓存 miss 时用 singleflight 合并同一分享的并发查询
- 仅一个请求打到数据库，其余等待结果
- 对热门分享使用更长 TTL 或不过期策略

**涉及文件：**
- `core/internal/cache/share_cache.go` — GetShareDetail 方法
- 新增 singleflight 依赖

---

### 2.6 前端 Element Plus 按需加载

**现状：** Element Plus 完整引入，打包后 CSS + JS 约 500KB+（gzip）。

**方案：**
- 安装 `unplugin-vue-components` 和 `unplugin-auto-import`
- Vite 配置中添加 Element Plus 按需导入解析器
- 移除全局 `import ElementPlus from 'element-plus'`
- 预计可减少 60-70% 的 Element Plus 包体积

**涉及文件：**
- `web/vite.config.ts` — 添加插件
- `web/src/main.ts` — 移除全局注册
- `web/package.json` — 新增开发依赖

---

### 2.7 删除生产环境调试代码

**现状：** `user-folder-children-logic.go:55` 中 `l.svcCtx.Engine.ShowSQL(true)` 全局开启 SQL 日志，影响所有查询性能。

**方案：** 删除此行，如需调试通过配置文件控制。

## 三、代码质量与工程化（P1）

### 3.1 单元测试体系

**现状：** 项目无任何单元测试。

**方案：**
- 使用 Go 标准 `testing` + `testify` 框架
- 优先覆盖核心业务逻辑：
  - 密码哈希与验证
  - JWT 生成与解析
  - 文件秒传匹配逻辑
  - 上传会话状态机
  - 批量操作（移动/删除/重命名）边界情况
  - 权限策略验证
- 集成测试使用 Docker Compose 启动 MySQL + Redis 真实依赖
- CI 中配置 `go test ./...` 自动运行

**目标覆盖率：** 核心 logic 层 ≥ 80%

---

### 3.2 soft-delete 过滤器遗漏修复

**现状：** 以下位置缺少 `deleted_at IS NULL` 过滤，可能导致软删除的数据被错误引用：

| 文件 | 位置 | 影响 |
|------|------|------|
| `file_common.go:38-39` | `resolveParentID` | 可上传到已删除的文件夹 |
| `file_common.go:57-58` | `resolveParentID` (by id) | 同上 |
| `user-folder-path-logic.go:47` | 起始文件夹查询 | 已删除文件夹出现在面包屑 |
| `user-folder-path-logic.go:69` | 父级遍历查询 | 同上 |
| `user-file-favorite-logic.go:28` | 收藏查询 | 可收藏已删除的文件 |

**方案：** 为上述所有查询添加 `AND deleted_at IS NULL` 条件。

---

### 3.3 统一错误处理

**现状：** Handler 层和 Logic 层错误处理风格不一致，有的直接返回 error，有的返回自定义 LogicError。

**方案：**
- 统一使用 `LogicError` 包装所有业务错误
- 定义标准错误码枚举（如 10001=参数错误、10002=认证失败、20001=文件不存在等）
- Handler 层统一通过 `httpx.ErrorCtx` 返回 JSON 格式：`{ "code": 10001, "message": "..." }`
- 前端根据 `code` 做差异化处理

## 四、现有功能优化（P2）

### 4.1 验证码过期时间延长

**现状：** 验证码 TTL 仅 60 秒（`core/internal/define/define.go:21`），用户经常来不及输入。

**方案：** 改为 300 秒（5 分钟），符合行业标准。

---

### 4.2 回收站自动过期清理

**现状：** 回收站中的文件永不自动清理，持续占用存储空间。

**方案：**
- 新增定时任务（每天凌晨 2 点执行）
- 自动永久删除回收站中超过 30 天的文件
- 删除时同步清理 `repository_pool`（当引用计数为 0 时）和 OSS 对象
- 用户可在设置中调整保留天数
- 在回收站页面显示 "X 天后自动删除" 提示

**涉及文件：**
- 新增 `core/cmd/recycle_cleaner/main.go` — 定时清理任务
- `web/src/components/FileTable.vue` — 回收站模式下显示剩余天数

---

### 4.3 存储空间配额管理

**现状：** 用户可无限量上传文件，存储成本不可控。

**方案：**
- `user_basic` 表新增 `storage_limit`（默认 5GB）和 `storage_used` 字段
- 上传前校验剩余空间是否足够
- 文件删除（永久删除）时回收空间
- 秒传也需计入用户已用空间
- API 新增 `GET /user/storage` 返回配额和使用量
- 前端侧边栏显示存储使用进度条

---

### 4.4 分享系统增强

**现有功能：** 基础链接分享、提取码、过期时间、下载权限。

**新增优化：**
- **分享浏览计数同步：** 修复 Redis 计数与数据库计数不一致的问题（`share-file-detail-logic.go:45-49`），以 Redis 为准，定期批量同步到数据库
- **文件夹分享：** 当前仅支持单文件分享，增加对文件夹的分享支持（展示目录结构）
- **批量分享：** 支持一次性创建多个文件的分享链接
- **分享统计：** 每日/每周/每月访问趋势图

---

### 4.5 文件搜索增强

**现有功能：** 文件名模糊搜索（LIKE）。

**新增优化：**
- **搜索去抖：** 前端输入 300ms 无操作后自动搜索，无需按回车
- **搜索高亮：** 搜索结果中高亮匹配的关键词
- **搜索历史：** 本地存储最近 10 条搜索记录
- **高级搜索：** 支持按文件大小范围、日期范围、文件类型组合筛选
- **全文检索（远期）：** 利用已有的 Elasticsearch 基础设施，对文档类文件建立全文索引

---

### 4.6 上传体验优化

**现有功能：** 单文件上传、断点续传、秒传、进度显示。

**新增优化：**
- **文件夹上传：** 支持 `webkitdirectory` 属性上传整个文件夹（自动创建目录结构）
- **拖拽上传：** DiskView 增加 drag-and-drop 区域
- **上传队列管理：** 添加 "清除已完成" 按钮（函数已存在于 store 但未绑定到 UI）
- **总体进度：** 显示 "3/7 个文件已上传" 的汇总进度
- **并发控制：** 支持配置同时上传的文件数（当前无限制）

### 5.3 文件标签系统

**方案：**
- 新增 `file_tag` 表和 `file_tag_mapping` 关联表
- 用户可为文件添加自定义标签（如 "工作"、"旅行"、"重要"）
- 支持按标签筛选文件
- 标签管理（创建/修改/删除/合并标签）

**数据模型：**
```sql
CREATE TABLE file_tag (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    identity VARCHAR(36) NOT NULL,
    user_identity VARCHAR(36) NOT NULL,
    name VARCHAR(50) NOT NULL,
    color VARCHAR(7) DEFAULT '#409EFF',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE file_tag_mapping (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    tag_identity VARCHAR(36) NOT NULL,
    file_identity VARCHAR(36) NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 5.4 用户中心

**现状：** 无用户信息管理页面，不能修改密码或头像。

**方案：**
- 新增 `GET /user/profile` — 获取个人信息
- 新增 `PUT /user/profile` — 修改昵称、头像
- 新增 `PUT /user/password` — 修改密码（需验证旧密码）
- 新增 `POST /user/avatar/upload` — 头像上传
- 前端新增用户中心页面

---

### 5.5 通知系统

**方案：**
- **站内通知：** 分享被访问、存储空间即将满、文件被系统清理等事件通知
- 通知存储在 MySQL `notification` 表
- 未读通知角标展示在导航栏
- 支持标记已读/全部已读
- 远期接入 WebSocket 实现实时推送

## 六、新增功能 — 进阶能力（P3）

### 6.1 离线下载

**方案：**
- 新增 `POST /download/task/create` — 创建下载任务（支持 HTTP/HTTPS 直链）
- 后端使用 goroutine 异步下载到 OSS
- 任务状态管理（排队中/下载中/已完成/失败）
- Redis 存储任务进度，前端轮询展示
- 初期仅支持 HTTP 直链下载，远期可扩展 BT/磁力
- 下载完成后自动保存到用户指定目录

---

### 6.2 文件评论 / 协作批注

**方案：**
- 新增 `file_comment` 表：`id, identity, file_identity, user_identity, content, parent_id, created_at`
- 支持对文件添加评论和回复
- 支持 @提及其他用户
- 评论列表展示在文件详情侧边栏

---

### 6.3 操作审计日志

**现有基础：** 已有 RabbitMQ + ES 异步日志管道。

**增强方案：**
- 新增 `audit_log` 表记录用户操作（上传/下载/删除/分享/修改权限等）
- 管理员可查看系统级审计日志
- 用户可查看自己的操作历史
- 对接已有的 ES 管道，支持按时间范围和操作类型检索
- API：`GET /admin/audit/logs`、`GET /user/audit/logs`

---

### 6.4 管理后台

**现状：** 仅有 Casbin RBAC 角色定义，无管理界面。

**方案：**
- 新增管理后台页面（`/admin`），仅 admin 角色可访问
- **用户管理：** 查看/禁用/删除用户、修改用户角色、调整存储配额
- **系统概览：** 用户数、文件数、存储总量、日活跃用户、上传/下载趋势图
- **分享管理：** 查看/禁止违规分享
- **系统设置：** 全局配置（最大上传大小、默认配额、允许的文件类型等）

---

### 6.5 文件加密存储

**方案：**
- 支持用户选择是否对文件进行加密存储
- 使用 AES-256-GCM 对称加密
- 加密密钥由用户密码派生（PBKDF2/Argon2）
- 加密文件在服务端不可解密（零知识架构）
- 加密元数据存储在 `repository_pool` 表新增字段

---

### 6.6 文件夹密码锁

**方案：**
- 支持对特定文件夹设置访问密码
- 访问加密文件夹需要先输入密码
- 密码验证后生成临时会话 Token（有效期 30 分钟）
- 适用于隐私空间场景

---

## 七、前端体验优化（P2）

### 7.1 深色模式

**现状：** 使用 CSS 变量（`:root`），技术上已具备切换基础。

**方案：**
- 定义 `:root[data-theme="dark"]` 变量覆盖集
- Element Plus 使用 CSS 变量覆盖暗色主题
- 导航栏添加主题切换按钮（太阳/月亮图标）
- 用户偏好存储到 localStorage
- 跟随系统主题（`prefers-color-scheme` 媒体查询）

---

### 7.2 拖拽交互

**现状：** 完全无拖拽支持。

**方案：**
- **拖拽上传：** DiskView 添加全局 drop zone，支持拖拽文件到页面任意位置上传
- **拖拽移动文件：** 支持拖拽文件到左侧文件夹树进行移动
- **拖拽排序：** 文件列表支持自定义排序（远期）
- 使用 `@vueuse/core` 的 `useDraggable` 或原生 Drag API

---

### 7.3 右键上下文菜单

**现状：** 文件操作仅通过行内按钮和下拉菜单触发。

**方案：**
- 文件列表区域右键弹出上下文菜单
- 菜单项：打开、下载、重命名、移动、收藏、分享、删除、属性
- 空白区域右键：新建文件夹、上传文件、粘贴、刷新
- 可使用 Element Plus 的 Dropdown 或自定义 ContextMenu 组件

---

### 7.4 键盘快捷键

### 7.5 骨架屏加载

**现状：** 仅 ShareView 有一处骨架屏，其他页面用 Element Plus 默认 loading。

**方案：**
- 文件列表加载时使用 `el-skeleton` 模拟表格行
- 侧边栏加载时使用骨架屏
- 面包屑加载时使用骨架屏
- 预览对话框加载中状态

---

### 7.6 空状态优化

**现状：** 所有空状态仅显示通用文字，无引导操作。

**方案：**
- 文件列表为空时显示自定义插图 + "上传第一个文件" 按钮
- 搜索无结果时显示 "未找到匹配文件，试试调整关键词"
- 回收站为空时显示 "回收站是空的，真干净！"
- 收藏列表为空时显示 "还没有收藏的文件" + 引导文案
- 分享列表为空时显示 "还没有分享记录" + "去分享" 按钮

---

### 7.7 文件图标升级

**现状：** 使用中文单字符 + 彩色背景作为文件图标。

**方案：**
- 引入 SVG 图标集（如 vscode-icons 或 file-icon 库）
- 按文件扩展名匹配对应图标
- 图片文件显示缩略图（OSS 图片处理能力生成缩略图 URL）
- 文件夹显示打开/关闭状态图标

---

### 7.8 无障碍优化（a11y）

**现状：** 仅 1 处 `aria-label`，基本无无障碍支持。

**方案：**
- 所有按钮添加 `aria-label`
- 侧边栏添加 `role="navigation"`
- 文件表格添加 `role="grid"` 和行级 `aria-label`
- 对话框焦点管理（打开时聚焦首个输入，关闭时恢复焦点）
- 操作结果通过 `aria-live` 区域通知屏幕阅读器
- 颜色对比度检查，确保符合 WCAG 2.1 AA 标准
- 添加 Skip to Content 跳转链接

### 7.9 国际化（i18n）

**现状：** 所有文案硬编码中文。

**方案：**
- 引入 `vue-i18n`
- 抽取所有硬编码文案到 `locales/zh-CN.json` 和 `locales/en.json`
- 导航栏添加语言切换器
- Element Plus 国际化配置
- 日期/数字格式本地化

## 八、新增功能 — 远期规划（P4）

### 8.1 多端同步客户端

**方案：**
- **桌面同步客户端：** 基于 Electron/Tauri 开发，监听本地文件夹变化自动同步
- **移动端 App：** 基于 uni-app 或 Flutter 开发 iOS/Android 客户端
- **WebDAV 协议支持：** 新增 WebDAV 接口，兼容第三方文件管理器

---

### 8.2 AI 能力集成

**方案：**
- **智能分类：** 调用 AI 接口自动为文件打标签
- **图片 OCR：** 提取图片中的文字，支持搜索
- **文档摘要：** 对文档文件生成摘要描述
- **相似文件推荐：** 基于文件内容和元数据推荐相关文件
- **智能搜索：** 自然语言搜索（"找一下上个月的会议记录"）

### 8.3 文件同步与冲突处理

**方案：**
- 使用向量时钟或 Last-Writer-Wins 策略处理并发编辑冲突
- 增量同步（rsync 算法）减少同步流量
- 离线编辑队列，重新上线后自动同步

---

### 8.4 多存储后端支持

**现状：** 仅支持阿里云 OSS。

**方案：**
- 抽象存储接口（`StorageProvider`）
- 实现多种存储后端：
  - 阿里云 OSS（已有）
  - 腾讯云 COS
  - AWS S3
  - 本地文件系统（开发/自部署场景）
  - MinIO（私有化部署）
- 配置文件中选择存储后端

---

### 8.5 团队空间 / 协作

**方案：**
- 新增 `team` 和 `team_member` 表
- 团队共享文件夹（团队成员可见）
- 团队内文件权限（查看/编辑/管理）
- 团队配额管理
- 邀请制加入团队

---

### 8.6 Webhook / 第三方集成

**方案：**
- 支持配置 Webhook，在文件上传/分享/删除时推送通知
- 对接企业微信/钉钉/飞书推送
- 开放 API 支持第三方应用接入
