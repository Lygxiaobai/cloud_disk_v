一、文件上传流程

1. 功能目标

  • 秒传
  • 分片上传
  • 断点续传
  • 暂停 / 继续
  • 失败重试
  • 上传进度展示
  • 剩余时间估算
  • STS 临时授权上传

2. 设计思路

上传流程拆成三段：上传初始化、前端直传 OSS、上传完成确认。

这样可以减少后端带宽压力，并充分利用 OSS 的分片上传能力，同时保留前端任务进度、暂停继续、重试和秒传判断能力。

二、文件预览流程

1. 功能目标

  • 支持图片、视频、音频、PDF、文本和其他文件预览

2. 设计思路

二进制媒体文件使用 OSS 临时签名 URL，文本类文件只读取前 4KB 以避免大文件整体加载，其他文件返回下载地址。

3. 流程图

+------------------------+
| 用户点击文件预览       |
+------------------------+
            |
            v
+-----------------------------------+
| 调用 GET /user/file/preview/:id   |
+-----------------------------------+
            |
            v
+-------------------------------+
| 后端根据扩展名判断文件类型    |
+-------------------------------+
     | 图片/视频/音频/PDF
     |-------------------------------> 生成 OSS 签名 URL
     |
     | 文本
     |-------------------------------> 读取前 4KB 文本
     |
     | 其他
     |-------------------------------> 返回下载地址
            |
            v
+------------------------+
| 前端弹窗展示预览       |
+------------------------+

三、最近文件流程

1. 功能目标

记录用户最近访问文件，便于快速定位常用内容。

2. 设计思路

后端在文件被成功预览后，将文件 identity 写入 Redis 的 ZSet，score 为最后访问时间戳。

+-----------------------------+
            |
            v
+------------------------+

+------------------------+

六、批量操作流程

1. 功能目标

  • 批量收藏
  • 批量移动
  • 批量删除

2. 设计思路

文件夹的批量操作不能只处理当前节点，而要处理整棵子树；同时移动、删除等操作必须放在事务里保证一致性。

3. 流程图

+------------------------+
| 用户勾选多个文件       |
+------------------------+
            |
            v
+------------------------+
| 选择批量操作类型       |
+------------------------+
    | 批量收藏
    |-----------------------------> 更新 is_favorite
    |
    | 批量移动
    |-----------------------------> 校验目标目录
    |                               校验不能移动到自己或子目录
    |                               批量更新 parent_id
    |
    | 批量删除
    |-----------------------------> 收集整棵子树
                                    批量写 deleted_at

七、回收站流程

1. 功能目标

  • 删除进入回收站
  • 回收站查看
  • 恢复
  • 彻底删除

2. 设计思路

回收站基于 deleted_at 软删除实现。恢复前会检查父目录状态，避免父目录未恢复但子文件先恢复导致层级错乱。

3. 流程图

八、整体总结

本次新增功能采用“前端直传 OSS + 后端发 STS + 资源池去重 + 软删除回收站 + Redis 最近文件”的组合方案，在保证功能完整的同时，也保留了后续继续扩展的空间。
  • 上传采用前端直传 OSS，减轻后端压力
  • 秒传通过 hash + size 命中资源池实现
  • 断点续传依赖 upload_session + checkpoint
  • 预览采用“签名 URL + 文本分段读取”组合方案
  • 最近文件采用 Redis ZSet 实现
  • 收藏采用字段方案实现，轻量直接
  • 批量操作与回收站统一基于子树处理和事务控制
